# shishengkai-brain

这是 **北京始盛开科技有限公司 / 始盛开品牌** 的项目 brain，用来沉淀公司命名、品牌叙事、视觉系统、项目方向与后续可执行文档。

`*-brain` 不是代码仓库，而是一个项目的 **「思考 — 沉淀 — 交付」系统**：把零散灵感、外部资料、与 AI 的对话，逐步整理成可执行的设计与文档，并作为项目落地实现的 **唯一事实来源（Single Source of Truth）**。

## 当前定稿内容

- [`Releases/始盛开的由来.md`](./Releases/始盛开的由来.md)：公司名称“始盛开”的来源、诗意解释与品牌叙事。
- [`Releases/LOGO设计.md`](./Releases/LOGO设计.md)：最终 LOGO 文件、设计定位、用色规则与使用原则。
- [`Releases/color-system/color-system.md`](./Releases/color-system/color-system.md)：始盛开颜色系统与 LOGO / UI 用色规则。

## 目录结构

```text
shishengkai-brain/
├── README.md          # 项目入口与说明
├── AGENTS.md          # 给 AI agent 的操作指南
├── Releases/          # 唯一事实来源：定稿的想法、设计、文档
├── Drafts/            # 草稿区：零散灵感、外部信息、与 AI 的对话
└── Archive/           # 归档区：已完成使命或被弃用的内容
```

| 目录 | 定位 | 放什么 | 原则 |
| --- | --- | --- | --- |
| **`Releases/`** | 唯一事实来源 | 定稿的项目说明、品牌叙事、视觉系统、设计方案、规范、路线图 | 默认「可照着做」，不写不确定的猜想 |
| **`Drafts/`** | 草稿 / AI 上下文素材 | 零碎想法、灵感、外部资料、与 GPT/Gemini 的对话记录、半成品梳理 | 允许混乱，但写清楚「为什么写它、卡在哪、下一步要补什么」 |
| **`Archive/`** | 历史归档 | 已完成使命或不再适用的文档 | 不删，但日常执行以 `Releases/` 为准 |

## 推荐工作流

1. **先丢进 `Drafts/`** —— 任何灵感、需求片段、外部文章、AI 对话先快速记录，不求完美。
2. **持续打磨草稿** —— 每次迭代都让它更「可执行」：补流程、示例、边界条件、风险、验证方式。
3. **达到可交付标准后移入 `Releases/`** —— 标准是：读者（人或 AI）不用追问也能照做。
4. **被替代或过时后移入 `Archive/`** —— 并在 `Releases/` 的新文档里写明替代关系 / 迁移指引。

## 如何让 agent 开始 / 项目接手

无论项目进行了多久，建议后来加入的维护者先从 `AGENTS.md` 和 `Releases/` 建立上下文。

**新 agent 接手本项目：**

```text
请先读 AGENTS.md，再读 Releases/（尤其“始盛开的由来”“LOGO设计”和“始盛开颜色系统”）。读完后，请用几句话复述你对始盛开品牌、当前定稿内容和视觉约束的理解；在我确认之前，不要执行任何具体任务，也不要修改仓库内容。
```

**整理新材料为草稿：**

```text
请先读 AGENTS.md。我要把下面这些材料整理进 shishengkai-brain：先不要改 Releases/，默认生成到 Drafts/，并在文档开头写清楚这份草稿的用途、未决问题和下一步需要确认的内容。
```
